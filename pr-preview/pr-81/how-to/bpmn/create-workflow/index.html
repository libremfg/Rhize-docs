Create workflows

This topic shows you how to use the BPMN UI to create event-driven workflows. To view the full list of elements and their parameters, refer to BPMN elements.

Prerequisites

To create a workflow, you’ll need:

  • An installed environment
  • Topics to publish and subscribe to
  • The ability to call the GraphQL API and understand JSON.

General procedures

Every workflow is different, but the overall procedure usually works like this:

  1. Create a start event:

    1. Select a single thin circle (Start) and drag it your process space.
    2. If the process is subscribing to a topic, in the Message > Name field, add the topic you want to subscribe to. For example, /orders/*/sap.

  2. Use tasks to process and filter data:

    If you want to…Use this BPMN activity
    Filter the contents of a messageJSONata transform.
    Call the GraphQL APIGraphQL query or mutation
    Call another BPMN workflowCall activity

  3. Create conditional flows and parallel processes with gateways. To establish a condition, write a JSONata expression in the arrow that leads to the conditional result.

  4. To run the workflow, toggle the Enabled switch.

Examples

These workflows demonstrate the minimum work needed to create some useful process.

Build an Alert

This workflow publishes an alert if a reading from an oven sensor exceeds 100 degrees.

  1. Create a message start event that subscribes to the correct topic (for example, oven1/sensors/thermometer).

  2. Connect the start event to a task. For Template, choose JSONata transform.

  3. In JSONata transform expression, write an expression to evaluate whether the temp exceeds 100 (for example: readings.temp > 100). Name the output variable highTemp.

  4. Connect the JSONata task to an exclusive gateway.

  5. Connect the gateway to an intermediate message event.

  6. In the flow that connects the gateway and the intermediate message event, add this condition: = $.highTemp.

  7. In the intermediate message event, add the topic to publish the alert to.

    In the Inputs field, add the message body. Use BODY for the local variable name and something like this for the Variable assignment value:

    {"message":"Temperature exceeded 100 degrees"}

  8. Add End events to the gateway and to the message event.

Write ERP material definition to database

This workflow receives information from the ERP system about a new material definition, extracts the values, then creates a new record in the database.

For simplicity, assume the order has only these properties:

{
  "MaterialDefinition": [
    {
      "Definition_ID": "RM-8001276",
      "Description": ["Sugar small bags 20kg"]
    }
  ]
}

  1. Create a start event that subscribes to the correct topic, for example orders.

  2. Connect the start event to a JSONata task.

  3. In the JSONata transform expression, filter for only the fields you need, and map them to the field names accepted by the API:

    $map($.MaterialDefinition, function($v) {
    {
        "id": $v.Definition_ID,
        "description": $v.Description
    }
})

    Name the output variable newMaterial.

  4. Connect the JSONata task to a GraphQL service task.

  5. Send $.newMaterial as the variable for the GraphQL mutation addMaterialDefinition. In Mutation body, define the operation and fields to return. In Variables, add the values for the payload.

    mutation newMaterial($input: [AddMaterialDefinitionInput!]!) {
  addMaterialDefinition(input: $input) {
    materialDefinition {
      iid
      id
      _createdOn
    }
  }
}

    {
  "input": $.newMaterial
}

    Finish the workflow with some error handling based on the response:

    1. Connect the GraphQL task to a gateway. Connect this gateway to two end events, one for error and one for success.
    2. In the flow to the Error, add an evaluation for whether $.Response.status is 200.
